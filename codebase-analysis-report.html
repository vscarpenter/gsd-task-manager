Section 1

Executive summary

One-line verdict A well-architected, well-documented codebase actively maintained against modern standards — but three large React components concentrate change risk, the sync push/pull engines lack direct unit tests, and nine transitive CVEs in the MCP server's dependency tree need patched overrides this sprint.

Top 5 risks

  1. Sync push/pull engines lack direct unit tests. lib/sync/pb-push.ts, lib/sync/pb-pull.ts, and lib/sync/pb-sync-helpers.ts are the LWW-conflict heart of multi-device data integrity and only get indirect coverage through pb-sync-engine.test.ts. High
  2. Nine transitive vulnerability advisories in the MCP server. Two High (fast-uri host confusion + path traversal), four Medium (hono <4.12.18), one Medium (ip-address XSS), one Medium (postcss). The existing override "hono": ">=4.12.14" is stale relative to the current advisory floor. High
  3. Three large components concentrate change risk. EditDrawer (298-line function), MatrixSimplified (279-line function), and SettingsPage (269-line function) each violate the <40-line function rule by 7–8×. They are also where five chronic edit-drawer.test.tsx timeouts live. High
  4. MCP server tests are excluded from the main Vitest run. vitest.config.ts:16 excludes packages/mcp-server/** — five test files for token validation, schemas, and write-ops never execute under bun run test, so pre-commit verification can ship MCP regressions silently. Medium
  5. Service worker accepts postMessage without origin verification. public/sw.js:47 handles a SKIP_WAITING command from any frame controlling the SW client. Practical exploit surface is small (same-origin scope, single command, no data return) but the missing event.origin check is a defensible-default omission. Medium

Top 3 strengths

  1. Disciplined layering. Zero components/** or app/** files import lib/db directly; the sync subsystem is a clean DAG with no cycles; only three any/as unknown escape hatches across all production TS source.
  2. Substantial, fast test suite. 1830 tests across 115 files, 1829 pass + 1 skip, 8.3s wall-clock. Coverage thresholds correctly set to 80/80/80/75. fake-indexeddb and localStorage polyfills handled in vitest.setup.ts.
  3. Documentation discipline. Eleven ADRs in docs/adr/, lessons captured in tasks/lessons.md, CLAUDE.md current with non-obvious dep rationale and PocketBase v0.23+ gotchas. .env.local and .env* gitignored; no secrets in tracked source.

Top 3 priorities for next sprint

  1. Clear the dependency advisories. Bump hono override to >=4.12.18; add fast-uri >=3.1.2, ip-address >=10.1.1, postcss >=8.5.14 overrides; bump direct deps next 16.2.6, pocketbase 0.26.9, zod 4.4.3. Drives bun audit to zero. Effort: Small.
  2. Fill the sync test gap. Direct unit tests for pb-push, pb-pull, pb-sync-helpers; delete the four duplicate test files in tests/sync/ that mirror tests/data/sync/; re-include packages/mcp-server/** in Vitest projects so MCP tests run under bun run test. Effort: Medium.
  3. Decompose the three large components. Extract useMatrixState, useEditDraft, useSettingsPageState hooks. Drops the largest functions in the codebase, unblocks the five chronic edit-drawer.test.tsx timeouts, and reduces the change-risk concentration in the matrix shell. Effort: Medium.
Section 2

Methodology & limitations

What was inspected

  • Standards source of truth: coding-standards.md v14.0 (file ≤350–400 lines, function ≤40 lines, nesting ≤3, ≥80% coverage, TDD, no any without justification).
  • Source review: 369 TS/TSX files across lib/, components/, app/, packages/mcp-server/src/ totaling 26,346 lines (per wc -l). Hand-read of vitest.config.ts, vitest.setup.ts, tsconfig.json, eslint.config.mjs, next.config.ts, public/sw.js, scripts/setup-pocketbase-collections.sh, the largest source files, and the entire lib/sync/ dependency graph.
  • Dependency evidence: bun audit in repo root and packages/mcp-server; bun outdated; review of package.json overrides block; license inspection of direct dependencies.
  • Architecture probe: grep sweeps for any/as unknown escape hatches; cross-file import maps to detect circular dependencies and layering violations between UI, hooks, and the data layer; module-fan-out counts on the largest components.
  • Test evidence: 115 test files in tests/, 5 more in packages/mcp-server/src/__tests__/; 1830 tests, 1829 pass + 1 skip, 8.3s wall-clock. Mock-pattern audit, brittle-selector grep, behavior-vs-implementation naming sample.
  • CI surface: .github/workflows/{security-audit,publish-mcp-server,claude-code-review,claude}.yml.

What was not inspected

  • Runtime behavior. No browser load test, no Playwright run, no IndexedDB integration sweep, no PocketBase production-server probe. All findings derive from static analysis and the existing test suite output — performance claims are confidence-flagged Medium where empirical signal is missing.
  • PocketBase collection schema drift. scripts/setup-pocketbase-collections.sh was read; the live collection at https://api.vinny.io was not introspected.
  • CloudFront edge configuration. cloudfront-function-*.cjs files were inspected for code shape only; deployed function state at the CDN was not verified.
  • Native & Worker workspaces. packages/native/ and worker/ were excluded — out of scope for the v9 web build.
  • Coverage percentage. bun run test -- --coverage was not executed during this audit (would require ~minutes; not gated on); coverage analysis is structural (which files have any test) rather than line-percentage.

Confidence calibration

Findings tagged High confidence are corroborated by direct source read plus a second signal (test-suite output, bun audit JSON + advisory database, or git-history cross-reference). Medium confidence indicates one source only or pattern-match without empirical execution. Low confidence findings are flagged for review rather than treated as conclusions.

7 · Performance 4 findings · 0 Critical · all M-confidence

Confidence note: this audit was static. Empirical Lighthouse / bundle-analyzer / IndexedDB-profiling measurement was not run; findings are pattern-based and flagged Medium confidence.

Mediumconf: MEffort SSync push throttle is fixed at 100ms — fine for typical, but no batching
lib/sync/pb-push.ts:111 · CLAUDE.md notes: "Push operations are throttled (100ms between requests) to avoid PocketBase 429 errors"

For users with hundreds of queued mutations after a long offline period, sequential 100ms requests = many seconds of push time. Batch via PocketBase's batch API (PB v0.23 supports it), or migrate to a single-request payload-list endpoint when available. Empirical first: instrument pushedCount/duration and ship telemetry before optimizing.

Mediumconf: MEffort SDashboard page cognitive complexity 25 likely renders all charts upfront
app/(dashboard)/dashboard/page.tsx:41

recharts is composable but every chart inside the dashboard is rendered on first paint. For below-the-fold charts (long-form trends, tag analytics), gate on IntersectionObserver or React.lazy(). Reduces TTI on the dashboard route. Empirical: run Lighthouse against the dashboard before/after.

Lowconf: MEffort SCapture parser regex risk under pathological input
lib/capture-parser.ts:17 · lib/task-links.ts:6

Already covered in Security. Performance dimension: even if the input length is bounded, a worst-case backtracking pass on a 2KB string is non-trivial work on the UI thread. Add a single String#includes("://") short-circuit before the regex pass to skip the typical no-URL case.

Infoconf: MVirtualization, memoization, and PWA caching look correct on inspection

@tanstack/react-virtual is wired into long task lists per CLAUDE.md · React Compiler is enabled in next.config.ts reducing manual useMemo need · public/sw.js caches only same-origin GET responses (line 63–67), correctly bypassing the PocketBase API · canvas-confetti is pinned to a specific version to keep the bundle reproducible. No obvious N+1 patterns found in lib/tasks or lib/sync.
