01 Executive Summary

8 / 10

Healthy, mature, disciplined

GSD Task Manager is in strong shape. Test coverage clears the 80% floor on lines, statements, and functions; the codebase has zero TODO/FIXME debt; structured logging replaces every console.* in app code; and architectural decisions are captured in eleven ADRs. The remaining work is mostly supply-chain hygiene (transitive vulns in the MCP SDK chain) and a small number of files that have crept past the 350-line guideline. None of it is structural.

Top 5 risks
  1. Transitive supply-chain vulns — 2 high, 6 moderate, 1 low (fast-uri, ip-address, postcss, hono) routed through @modelcontextprotocol/sdk.
  2. Branch coverage 78.27% — narrowly below the 80% floor in coding-standards.md §3; concentrated in dashboard/matrix page shells.
  3. ARCHITECTURE.md drift — last edited 2026-03-20, predates the v9 single-matrix refactor (ADR 0011, 2026-04-28). New contributors see a stale system map.
  4. Dormant components/command-palette/ (347 LoC, four files) is on disk but not wired into the v9 shell. Either re-wire or move behind a feature flag with a removal date.
  5. Two 0%-covered route shells (app/(dashboard)/dashboard/page.tsx, app/(matrix)/page.tsx) lower aggregate confidence and hide regressions in shell wiring.
Top 3 strengths
  1. Discipline shows in the metrics — 0 TODO/FIXME, 0 console.* in app code, 2 justified any casts across 27.7K LoC.
  2. Verification is real — five CI workflows (security audit nightly + SonarCloud + Claude review + MCP publish + bot triggers), fake-indexeddb integration, ~2,398 test cases across 120 files, 299 explicit mocks.
  3. Decisions are written down — eleven numbered ADRs covering IndexedDB-only, PocketBase sync, LWW resolution, PWA, MCP, smart views, and the v9 refactor. Reviewers and future agents have context.
Top 3 priorities for the next sprint
  1. Bump the MCP SDK (or pin fast-uri/ip-address/hono via overrides) to clear both High advisories. The override pattern is already established in package.json.
  2. Refresh ARCHITECTURE.md to reflect the v9 single-matrix shell, the modular components/matrix-simplified/* layout, and the Inkwell rollout. Cross-link ADR 0011.
  3. Cover the route shells with smoke tests (render <Dashboard /> with a stub task list) — picks up branch coverage past 80% with one PR.

03 Detailed Findings

Dimension 03 · Security & supply chain

High Conf · High Two high-severity advisories reach via the MCP SDK transitive chain
File · bun.lockEffort · Small (under 1 day)

fast-uri ≤ 3.1.1 has two open advisories: host confusion via percent-encoded authority delimiters (GHSA-v39h-62p7-jpjc) and path traversal via percent-encoded dot segments (GHSA-q3j6-qgpj-74h6). It reaches the lockfile through @modelcontextprotocol/sdk and through eslint → @eslint/eslintrc → ajv → fast-uri.

Recommendation. The repo already uses overrides for transitive CVE pins. Add one entry and run bun install:

"overrides": {
  ...,
  "fast-uri": ">=3.1.2"
}

Verify with bun audit --audit-level=high in CI; the existing .github/workflows/security-audit.yml already gates on high.

Medium Conf · High Six moderate advisories cluster around hono, postcss, and ip-address
File · bun.lockEffort · Small (under 1 day)

Four of the six are hono findings (JSX SSR style injection, cache vary leakage, body-limit bypass, JSX tag injection). All reach via @modelcontextprotocol/sdk; none are exposed to users — the MCP server is local-stdio only — but they pollute the audit baseline. postcss < 8.5.10 is a build-time dep with a stringify XSS that does not affect runtime, and ip-address is build-time too.

Recommendation. Bump @modelcontextprotocol/sdk to the latest minor (it pulls clean transitive deps) or add "hono": ">=4.12.18", "ip-address": ">=10.1.1", and "postcss": ">=8.5.10" to the overrides block. The postcss override is already at 8.5.10 — the audit hit suggests a transitive resolver disagrees. Re-pin to ">=8.5.10" to be certain.

Low Conf · High No dynamic code execution, no SQL string concatenation, no raw HTML sinks

Grep across lib/ · components/ · app/ · packages/mcp-server/src/ finds zero usages of unsafe HTML injection APIs and zero dynamic-code-execution constructs. Storage is IndexedDB through Dexie, which uses parameterized queries by construction. PocketBase access is via the official SDK; no string-concatenated filters appear in lib/sync/. This is a strength worth preserving — surface in PR review whenever the codebase grows.

Dimension 02 · Test quality & coverage

Medium Conf · High Branch coverage 78.27% — narrowly below the 80% floor

The aggregate is 1.73 points short. The shortfall is concentrated in two files:

The two route shells are not loaded by any test. Adding a one-shot render test (render(<Page />) with stub data) for each lifts the aggregate over 80% in one PR. The dashboard page in particular has 21 functions at 0% — losing it to a regression would be invisible.

Low Conf · Medium ~2,398 tests with 299 explicit mocks — healthy unit-vs-integration balance

120 test files, organised by domain (tests/data/, tests/ui/, tests/sync/, tests/utils/, packages/mcp-server/src/__tests__/). fake-indexeddb is auto-imported in vitest.setup.ts, so persistence-layer tests exercise real Dexie code paths against an in-memory backend rather than mocking db.ts. vi.mock('pocketbase') is used at the network boundary only. This is the correct shape — real code below the seam, mocks only at the I/O boundary — and matches coding-standards.md §3 guidance on independent tests.

Watch-list. The 590-line tests/data/sync/queue.test.ts and 837-line tests/data/notifications.test.ts are large; the size is data-driven (parametrised cases) rather than coupling, but split if either grows past 1,000 lines.

Low Conf · Medium No spec-driven test stubs in tasks/spec.md

coding-standards.md §1 / Spec-Driven Development requires "Test Stubs: Draft test function names (empty bodies) mapping to each criterion. Shipped with the spec." There is no tasks/spec.md — only tasks/todo.md, tasks/lessons.md, and dated review notes. For a mature codebase shipping at v9.1.1, this is acceptable for incremental work, but the next non-trivial feature should adopt the spec.md + test-stub pattern from the start. Marked as assumption — verified absence of tasks/spec.md; lower-confidence on whether the team uses an out-of-tree spec.

Dimension 07 · Performance

Low Conf · Medium Hot-path patterns look correct — virtual scrolling, batched PB lookups, throttled push

@tanstack/react-virtual is wired for the matrix list, fetchRemoteTaskIndex() in lib/sync/pb-sync-engine.ts batches remote IDs to avoid the N+1 fetch pattern, and push operations are throttled at 100 ms to dodge PocketBase's 429s. dexie-react-hooks's useLiveQuery handles the live-update path without manual subscription bookkeeping. The React 19 + babel-plugin-react-compiler combo removes the need for hand-rolled useMemo/useCallback on most components.

Not directly verified. No bundle-size budget or Lighthouse run was performed in this audit; if a regression appears, run chrome-devtools-mcp's performance_start_trace against http://localhost:3000 to capture LCP/INP.

Dimension 08 · Error handling & observability

Low Conf · High Structured logger replaces every console.* in app code

lib/logger.ts (284 LoC) defines createLogger(context) with environment-aware level gating and secret sanitisation. Grep finds zero raw console.log/warn/error in lib/, components/, or app/ outside of logger.ts itself — every diagnostic goes through a named context (e.g. const logger = createLogger("DB")). This satisfies coding-standards.md §3 / Error Handling and §10 red-flags.

Low Conf · Medium Toast-based user-error surface (sonner) replaces window.alert

The April 2026 standards audit captured a lesson: window.alert() is not accessible. The codebase now standardises on toast.error() from sonner, which uses ARIA live regions. tasks/lessons.md records the rationale; treat this as a "do not regress" rule and add a hook-level grep against window\.alert\( if it ever recurs.

Dimension 11 · Configuration & type safety

Low Conf · High Two any usages, both justified 
components/install-pwa-prompt.tsx:40
   (window.navigator as any).standalone === true;

packages/mcp-server/src/tools/handlers/index.ts:63
   const typedArgs = validateToolArgs(name, args) as any;

The first is the legitimate iOS-specific navigator.standalone escape (no DOM lib type for it). The second is a post-validation cast — the discriminator is checked by Zod immediately above, so the any is operator-safe. Add a one-line justification comment per coding-standards.md §2 / Type Safety and they are squared away. The MCP cast in particular could become z.infer<typeof schema> if the schema map is keyed by tool name.

Low Conf · High Schema validation at every boundary (Zod) — import uses .strip() intentionally

lib/schema.ts defines Zod schemas for every persisted entity. Import paths use .safeParse() + .strip() to accept legacy exports with extra fields (e.g. vectorClock from the retired Cloudflare sync), while export paths use .strict() to keep outgoing data clean. This asymmetric pattern is explicitly documented in both CLAUDE.md and tasks/lessons.md — the right shape for a sync-aware schema.

05 Methodology & Limitations

What was inspected

Static analysis of the working tree at HEAD f98fd21 on main: file-size and structural sweep of lib/, components/, app/, packages/mcp-server/src/, and tests/; bun audit against bun.lock; coverage extracted from the most recent coverage/coverage-summary.json (vitest + v8); grep sweeps for tech-debt markers (TODO / FIXME / HACK), unsafe HTML / dynamic-code constructs, raw console.* usage, any escape hatches; verification of the eleven ADRs in docs/adr/ and the five workflows in .github/workflows/; cross-reference against coding-standards.md v14.0 for every finding.

What was NOT inspected

  • Runtime behavior. No app was launched, no user flows exercised, no Lighthouse / a11y / Playwright trace captured. Performance findings are based on code-shape reading, not measurement.
  • Build output. The out/ static export was not analysed for bundle size or chunking.
  • PocketBase server. The self-hosted backend at api.vinny.io and its admin config (collection rules, OAuth provider setup) were not reviewed; only the client-side sync code was.
  • MCP runtime. The MCP server was not connected to a live Claude Desktop instance; tool handlers were read but not exercised.
  • Cross-browser support. No Safari / Firefox / mobile browser checks performed on the Inkwell rollout.
  • Generated files. node_modules/, .next/, out/, coverage/, lockfiles, and tsconfig.tsbuildinfo were excluded from LoC and pattern sweeps per the original brief.

Confidence calibration

High findings are anchored to a verified file path, line number, advisory ID, or coverage data point. Medium findings rely on aggregate signals (e.g. mock-count grep) where a sampled spot-check matched the aggregate. Low findings — none in this report — would be reserved for inferences based purely on naming conventions or file presence. All assumptions are flagged inline in italics within the relevant finding body.
