The big picture

Where data lives

Source of truth is local IndexedDB. PocketBase is a mirror for multi-device fan-out, not the database of record.

What triggers a sync

UI mutations enqueue an op. Background timer, app focus, and the sync button all call coordinator.requestSync().

Conflict resolution

Last-write-wins on client_updated_at. Newer timestamp wins; ties go to remote.

Realtime

SSE subscription delivers other devices' writes immediately. Periodic sync is a safety net.

Auth flow

The PocketBase SDK does all the heavy lifting. The user clicks “Sign in with Google” → authWithOAuth2 opens a popup, runs the OAuth dance, stashes the JWT in localStorage, and keeps it refreshed. Sync code just asks the singleton whether it's authenticated and grabs the user id.

lib/sync/pb-auth.ts:41 
// Runtime allowlist — type erasure means callers could pass anything
const ALLOWED_OAUTH_PROVIDERS = new Set<OAuthProvider>(['google', 'github']);

export async function loginWithProvider(provider: OAuthProvider): Promise<AuthState> {
  if (!provider || !ALLOWED_OAUTH_PROVIDERS.has(provider)) {
    throw new Error(`OAuth provider not allowed: ${String(provider)}`);
  }

  const pb = getPocketBase();
  // SDK handles popup, code exchange, token storage, and refresh
  const authData = await pb.collection('users').authWithOAuth2({ provider });

  return {
    isLoggedIn: true,
    userId: authData.record.id,   // becomes the `owner` FK on every task row
    email: authData.record.email,
    provider,
  };
}

Why the allowlist isn't theatre

Any provider configured server-side would otherwise be callable from the browser console. We only accept the two we ship UI for.

The singleton

lib/sync/pocketbase-client.ts:18 
export function getPocketBase(): PocketBase {
  if (!pbInstance) {
    pbInstance = new PocketBase(ENV_CONFIG.pocketBaseUrl);
    // CRITICAL: SDK cancels in-flight requests by default when a new one
    // fires on the same collection. Our push loop would tear itself apart.
    pbInstance.autoCancellation(false);

03

Local mutation → push

Every task mutation does two things in order: write to Dexie, then enqueue a sync op. The queue is durable (it's an IndexedDB table), so offline edits survive a reload and get flushed on the next successful sync.

lib/sync/pb-push.ts:97 
export async function pushLocalChanges(): Promise<PushResult> {
  const ownerId = getCurrentUserId();
  if (!ownerId) return { ..., authenticated: false };  // no-op when logged out

  const pending = await getSyncQueue().getPending();
  if (pending.length === 0) return { ..., authenticated: true };

  // Pre-fetch all remote ids in ONE request — avoids N "does this exist?" lookups
  const { index: remoteIndex, fetchSucceeded } = await fetchRemoteTaskIndex(ownerId);

  for (const item of pending) {
    try {
      await pushSingleItem(item, remoteIndex, fetchSucceeded, ownerId, deviceId);
      pushedCount++;
      if (pushedCount + failedCount < pending.length) {
        await delay(THROTTLE_MS);   // 100ms gap → stay under PB's 429 threshold
      }
    } catch (error) {
      const errorCode = sanitizeSyncError(error);   // stable code only — raw 4xx bodies can echo task text
      await queue.recordAttemptFailure(item.id, errorCode);
      if (isRateLimitError(error)) break;            // 429 → abort, don't amplify the storm
    }
  }
}

Failed items aren't deleted

After SYNC_CONFIG.MAX_RETRIES attempts, a queue item flips to status: 'failed' and stays in db.syncQueue. Failed items are visible in the sync history UI for manual recovery; getPending() filters them so they don't clog the push loop.

Pull & last-write-wins

Pull uses a cursor (lastSyncAt) so we only fetch what changed since last time. The filter is the most subtle part — see Gotcha #1 about why we can't use PocketBase's own updated column.

lib/sync/pb-pull.ts:83 
export async function pullRemoteChanges(lastSyncAt: string | null) {
  const ownerId = getCurrentUserId();
  if (!ownerId) return { ..., authenticated: false };

  assertSafeRecordId(ownerId, 'ownerId');           // reject filter-injection attempts

  let filter = `owner = "${escapeFilterValue(ownerId)}"`;
  if (lastSyncAt) {
    filter += ` && client_updated_at > "${escapeFilterValue(lastSyncAt)}"`;
  }

  const records = await pb.collection('tasks').getFullList({
    filter,
    sort: '-client_updated_at',   // NOT `-updated` — system fields aren't sortable in PB v0.23+
  });

  // LWW merge into Dexie. Remote wins on ties (>=).
  for (const record of validRecords) {
    const localTask = localTaskMap.get(taskId);
    const remoteTask = pocketBaseToTaskRecord(record, localTask ?? null);
    if (!localTask || new Date(remoteTask.updatedAt) >= new Date(localTask.updatedAt)) {
      await db.tasks.put(remoteTask);
    }
  }

  await reconcileDeletedTasks(ownerId);   // drop local rows the server no longer has
}

The mapper layer is the single place where camelCase ↔ snake_case lives, and it also preserves device-local fields (notification dismissal state, snooze) so a sync doesn't undo “I dismissed that ping on this device”:

lib/sync/task-mapper.ts:122 
export function pocketBaseToTaskRecord(record, existingLocal) {
  const parsed = pbTaskRecordSchema.safeParse(record);   // fail closed on malformed remote payloads
  if (!parsed.success) return null;

  const r = parsed.data;
  return {
    id: r.task_id, title: r.title, /* … */
    notificationSent:    existingLocal?.notificationSent ?? false,    // per-device
    lastNotificationAt:  existingLocal?.lastNotificationAt,           // per-device
    snoozedUntil:        existingLocal?.snoozedUntil,                 // per-device
  };
}
05

Realtime (SSE)

The pull loop covers correctness; the SSE subscription covers latency. PocketBase pushes every write on the tasks collection to all connected clients; we filter out our own writes by device_id so we don't ping-pong our changes back into Dexie.

lib/sync/pb-realtime.ts:38 
const realtimeRecordShape = z.object({
  device_id: z.string(), task_id: z.string(), owner: z.string(),
});

export async function subscribe(deviceId: string) {
  currentDeviceId = deviceId;
  unsubscribeFn = await pb.collection('tasks').subscribe('*', handleRealtimeEvent);
}

async function handleRealtimeEvent(event) {
  const parsed = realtimeRecordShape.safeParse(event.record);
  if (!parsed.success) return;   // fail closed on weird shapes from a compromised server
  const record = parsed.data;

  // Echo filter — require BOTH non-empty so cold-start (null deviceId) doesn't swallow everything
  if (record.device_id && currentDeviceId && record.device_id === currentDeviceId) return;
  if (record.owner !== getCurrentUserId()) return;

  await applyRemoteChange(event.action, event.record);   // same LWW merge as pull
}

The coordinator

Anyone can call requestSync() — the sync button, the background timer, the health monitor, the app on focus. The coordinator collapses concurrent requests into one in-flight sync plus at most one queued follow-up, with user-priority overriding auto-priority.

SignalWhereWhat happens
Click “Sync now” sync-provider.tsx:172 requestSync('user') — bypasses backoff
Background timer background-sync.ts requestSync('auto') — gated by retry backoff
Local mutation tasks/crud/* Enqueue only; sync happens on next tick
SSE event pb-realtime.ts Apply directly via applyRemoteChange()
Health monitor health-monitor.ts Logs issues, does not auto-sync
07

Gotchas

System fields are unsortable / unfilterable in PocketBase v0.23+

You cannot sort: '-updated' or filter: 'updated > X' — the query parser refuses. Every timestamp the sync code touches is client_updated_at, a custom field set from task.updatedAt. Same applies to indexes: custom indexes can't reference updated/created.

Admin endpoint moved

Use /api/collections/_superusers/auth-with-password, not the legacy /api/admins/auth-with-password. If you script collection setup, hit the new one.

_pb_users_auth_ is not a real collectionId

Don't use that placeholder as the collectionId of a relation field — it silently breaks. Either use a plain text field for the owner FK (what we do), or look up the actual users-collection id and substitute it.

SDK auto-cancellation will tear apart a push loop

The default behaviour cancels any in-flight request when a new one fires on the same collection. We call pb.autoCancellation(false) in the singleton — don't re-enable it.

100ms throttle is load-bearing, not paranoia

PocketBase returns 429s under bursts. We sleep THROTTLE_MS between push items, and if a 429 still slips through we abort the push loop early rather than grind through the queue and amplify the response.

LWW uses >=, not >

On a timestamp tie, remote wins. Deliberate — if two devices write at the same millisecond, whichever made it to the server first becomes truth, the same invariant the pull cursor relies on.

Device-local fields are not synced through

notificationSent, lastNotificationAt, snoozedUntil are preserved from the existing local row when a remote update lands. Adding a new per-device field means editing pocketBaseToTaskRecord in task-mapper.ts — not just the schema.

Echo filter needs both sides non-empty

A naïve record.device_id === currentDeviceId would treat a cold-start race (where currentDeviceId is still null) as “this is my echo, skip it” and drop every legitimate event. The actual check requires both to be truthy.

Deletes are reconciled, not pushed individually

Pull ends with reconcileDeletedTasks(): any local task missing from the remote index and with no pending sync op gets deleted locally. If the remote index fetch fails, reconciliation is skipped — better to keep a stale row than to delete real data based on an incomplete view.

Raw error messages from PB must never be persisted

4xx responses can echo task content (titles, descriptions) back. Push errors run through sanitizeSyncError() to a stable code before they hit queue.lastError or any log. Do the same if you add new write paths.

Local dev tip

Set NEXT_PUBLIC_POCKETBASE_URL=https://api.vinny.io in .env.local to test against production OAuth. A local PocketBase at 127.0.0.1:8090 needs its own provider setup — Google/GitHub apps need the right redirect URIs registered.

08

If you only remember four things

TL;DR

  1. Local Dexie is the source of truth. PocketBase is a fan-out mirror.
  2. Every timestamp comparison uses client_updated_at. Never trust system fields.
  3. Mutations enqueue durably; the coordinator decides when to flush.
  4. Realtime SSE applies the same LWW merge as the pull loop — one merge rule, two places.
01 / Source of truth

Treat Dexie as the database. PocketBase is replication infrastructure.

02 / Custom timestamps

Always client_updated_at. The updated system field is off-limits for filter/sort.

03 / Durable queue

Mutations enqueue locally; the coordinator owns scheduling and backoff.

04 / One merge rule

Pull and realtime apply the same LWW (>=) merge. Two surfaces, identical behavior.
