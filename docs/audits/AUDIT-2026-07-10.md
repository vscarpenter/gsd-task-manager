# Application Audit — 2026-07-10

## Executive summary

The application has a strong client-side security baseline: PocketBase owner rules are scoped, import data is schema-validated, service-worker caching excludes authenticated and API traffic, and no live credentials were found in the sampled tracked files or configuration history. It is not ready to ship, however, because a confirmed first-run state bug leaves new users trapped behind onboarding, while the same modal also fails core keyboard-modal requirements; seven additional state-integrity, sync, accessibility, and responsive-layout defects should be scheduled behind those blockers.

**Release recommendation:** **Do not ship.** The first-run dialog cannot be dismissed without a reload or navigation, blocking the primary product path for every new browser profile. The modal's focus behavior is also a WCAG 2.2 AA release blocker for keyboard users.

## Phase 0 — Reconnaissance

- **Stack:** Next.js 16 static-export App Router, React 19, TypeScript, Tailwind CSS 4, Radix primitives, Bun workspaces, Dexie/IndexedDB local persistence, Zod validation, and an optional Node.js MCP server. Optional cloud sync uses PocketBase with Google/GitHub OAuth; Sentry is opt-in.
- **Entry points:** Static routes `/`, `/dashboard`, `/archive`, `/sync-history`, `/install`, `/about`, and `/settings`; local JSON import/export; IndexedDB task mutations; service-worker `fetch`, `notificationclick`, `periodicsync`, `push`, and update-message handlers; PocketBase OAuth, realtime events, and pull/push synchronization; and stdio MCP read/write tools.
- **Trust boundaries:** User-entered task/import content crosses into IndexedDB and optionally PocketBase; OAuth tokens cross from PocketBase into browser storage; PocketBase collection rules enforce remote ownership; MCP environment configuration supplies an endpoint and bearer token to a local Node process; third-party boundaries include PocketBase, OAuth providers, and opt-in Sentry.
- **Crown jewels:** Task titles, descriptions, tags, subtasks, dependencies, and time data; PocketBase JWTs and account identity; owner-scoped remote task records; backup exports; and account-deletion controls.
- **Deep flows:** First-run/local task CRUD and import; recurring completion and archive behavior; PocketBase OAuth plus push/pull/realtime reconciliation; MCP reads/writes; and service-worker caching/update behavior.
- **Not applicable or materially reduced:** There is no SQL database, server-rendered application endpoint, payment flow, webhook receiver, or server-side file upload, so SQL injection, checkout/payment abuse, conventional CSRF, and upload-path traversal do not apply. SSRF is limited to the explicitly user-configured local MCP endpoint and is constrained to HTTPS or loopback HTTP. Rate limiting is primarily a PocketBase deployment concern rather than an application route concern.

## Findings

### High

#### Reliability — First-run onboarding cannot be dismissed in the current render

**Severity:** High

**Category:** Reliability

**Location:** `components/onboarding/onboarding-gate.tsx:34-59`, `OnboardingGate`; `tests/ui/onboarding-gate.test.tsx:20-45`

**Issue:** The automatic open state is derived from `useSyncExternalStore(noopSubscribe, ...)`, but closing only writes `localStorage` and calls `setForceOpen(false)`. On a first-run automatic open, `forceOpen` is already false, the no-op subscription never publishes the changed storage snapshot, and React has no state change that can re-render the gate.

**Impact:** New users who press **Skip**, Escape, or **Start using GSD** remain behind the modal and cannot use the primary application until they navigate or reload. This affects every fresh browser profile and makes the first-run path appear broken.

**Evidence:** `open` is computed as `forceOpen || (!suppressed && !seen)` at line 47. `close()` writes the seen key and sets the already-false `forceOpen` at lines 49-52. Existing gate tests verify opening and replay but never click a close control and assert that the mounted gate disappears. A browser run against the static export confirmed that clicking **Skip** wrote the seen flag while the dialog remained mounted; reloading then removed it.

**Reproduction:** In a fresh browser profile, visit `/about/`, choose **Open App**, and click **Skip** in the welcome dialog. Inspect the DOM: the `Get the right things done.` dialog remains present. Reload `/`; it is then absent because the stored flag is re-read.

**Recommended fix:** Make the storage-backed state observable or maintain an explicit dismissed state that `close()` changes. Add a gate-level regression test that clicks **Skip**, Escape, and **Start using GSD** on an automatic first-run open and asserts that the dialog unmounts without navigation or reload.

**Confidence:** Confirmed

#### Accessibility — Onboarding modal does not contain or restore focus

**Severity:** High

**Category:** Accessibility

**Location:** `components/onboarding/onboarding.tsx:105-206`, `OnboardingDialog`; `components/settings/about-section.tsx:40-49`, replay trigger

**Issue:** The hand-rolled `role="dialog"` sets initial focus and listens for Escape, but it does not trap focus, mark the background inert, capture the invoking element, or restore focus after close. `aria-modal="true"` does not implement those behaviors.

**Impact:** Keyboard users can leave the modal and operate background controls while it is announced as modal, and users who replay the tour from Settings lose their place when it closes. This is a widespread modal-navigation failure and blocks WCAG 2.2 AA keyboard/focus conformance.

**Evidence:** The effect at lines 110-120 only registers a window keydown handler and calls `closeRef.current?.focus()`. The dialog at lines 125-206 contains no tab-cycle logic or focus-scope primitive, and no code stores/restores `document.activeElement`. In the browser run, replaying from Settings and closing left `document.activeElement` on `BODY`, not on the **Replay welcome tour** trigger.

**Reproduction:** On `/settings/`, activate **Replay welcome tour** by keyboard, close the tour, and inspect focus; it is not returned to the trigger. While open, continue tabbing past the dialog's last action and observe that no focus-containment mechanism exists.

**Recommended fix:** Replace the hand-rolled modal with the project's Radix dialog/focus-scope primitive, or implement focus containment, background inerting, and restoration to the exact invoking control. Add keyboard tests for forward/reverse tab wrapping, Escape, background isolation, and focus restoration.

**Confidence:** Confirmed

### Medium

#### Security — MCP sync status discloses the configured PocketBase endpoint

**Severity:** Medium

**Category:** Security

**Location:** `packages/mcp-server/src/tools/handlers/read-handlers.ts:18-35`, `handleGetSyncStatus`; `packages/mcp-server/src/api/client.ts:5-18`, `redactPocketBaseHost`; `packages/mcp-server/src/server/config.ts:27-53`, endpoint validation

**Issue:** `get_sync_status` serializes `config.pocketBaseUrl` directly into MCP tool output even though the MCP client module defines a redaction function specifically to keep private self-hosted endpoints out of chat logs, screenshots, and bug reports. HTTPS configuration also accepts URL userinfo, paths, queries, and fragments, all of which are echoed.

**Impact:** Invoking a routine status tool can place a private hostname, IP, port, deployment path, or URL-embedded secret into the model transcript and any downstream conversation logging. It does not expose the bearer token field, but it violates the endpoint-confidentiality boundary the package already documents.

**Evidence:** Line 25 assigns `pocketBaseUrl: config.pocketBaseUrl`, and line 32 returns the full result as text. By contrast, `redactPocketBaseHost()` replaces hostname and port for network errors because those messages may be shared. `isSafePocketBaseUrl()` accepts every syntactically valid HTTPS URL without rejecting credentials or query material.

**Reproduction:** Configure the MCP server with `GSD_POCKETBASE_URL=https://private.example:8443/pb?key=example`, invoke `get_sync_status`, and inspect the returned text; the full URL is present.

**Recommended fix:** Return only a non-sensitive connection descriptor or pass the value through the existing redaction helper. Reject URL credentials and unnecessary query/fragment material during configuration validation, and add a handler test asserting that the real host and URL secrets never appear in tool output.

**Confidence:** Confirmed

#### Race Condition — Concurrent recurring completions create duplicate future tasks

**Severity:** Medium

**Category:** Race Condition

**Location:** `lib/tasks/crud/toggle.ts:13-49,81-113`, `toggleCompleted` and `createAndQueueRecurringInstance`; `components/task-card/task-card-header.tsx:87-107`; `components/matrix-simplified/index.tsx:122-128`

**Issue:** Completing a recurring task is a read-create-update sequence with no transaction, compare-and-set, deterministic recurrence key, or in-flight UI guard. Two clicks can both read the same incomplete record and each create a different random-ID next instance before either updates the original.

**Impact:** A double-click, slow device, or two concurrent callers can create duplicate recurring tasks and duplicate remote sync operations. Users then receive duplicate work and reminders, and cleanup is manual.

**Evidence:** `toggleCompleted()` reads at line 19, creates the next instance at lines 28-30, and only later updates the original at line 36. `buildRecurringInstance()` generates a fresh ID at line 107. The completion button has no disabled/pending guard, and `handleToggle()` awaits the operation without tracking an in-flight task. Existing recurrence tests exercise one call only.

**Reproduction:** Create a recurring incomplete task and invoke `Promise.all([toggleCompleted(id, true), toggleCompleted(id, true)])` before either call completes. Both calls can observe `completed === false`; query IndexedDB and find two new instances with the same parent and due date.

**Recommended fix:** Perform the read, completion transition, and recurrence creation in one Dexie transaction with an idempotency key or conditional state check. Disable the completion action per task while pending, and add a concurrent-call regression test.

**Confidence:** High Confidence

#### Reliability — Task mutation and sync-queue writes can commit only half an operation

**Severity:** Medium

**Category:** Reliability

**Location:** `lib/tasks/crud/create.ts:41-55`; `lib/tasks/crud/update.ts:42-54`; `lib/tasks/crud/delete.ts:15-34`; `lib/tasks/crud/toggle.ts:28-49,81-95`; `lib/tasks/crud/helpers.ts:32-42`

**Issue:** Local task/dependency writes and their required sync-queue writes are separate IndexedDB transactions. If the queue add fails after the task mutation succeeds, the public operation throws even though local state has changed and there is no durable operation to propagate remotely.

**Impact:** With sync enabled, storage pressure, transaction aborts, or IndexedDB faults can produce silent local/remote divergence. A failed queued delete can later resurrect from the server; creates and updates can remain device-local indefinitely; recurring completion can leave a child created while the parent operation reports failure.

**Evidence:** Create writes `db.tasks.add()` at line 46 before `enqueueSyncOperation()` at lines 50-55; update and delete follow the same ordering. The helper performs a separate `syncQueue.add()` through `queue.enqueue`. The import path demonstrates the intended atomic model by opening a Dexie transaction over both `db.tasks` and `db.syncQueue` at `lib/tasks/import-export.ts:165-190`.

**Reproduction:** Enable sync, stub `syncQueue.add` to reject, and call create, update, or delete. The caller receives a failure, but querying `db.tasks` shows the mutation committed and querying `db.syncQueue` shows no matching operation.

**Recommended fix:** Wrap each logical mutation in one Dexie transaction spanning every affected task/dependency table and `syncQueue`, then schedule background sync only after commit. Add failure-injection tests proving that queue failure rolls back the corresponding local mutation.

**Confidence:** High Confidence

#### Reliability — Permanently failed queue items prevent remote deletions forever

**Severity:** Medium

**Category:** Reliability

**Location:** `lib/sync/queue.ts:23-25,53-75,102-130`; `lib/sync/pb-sync-engine.ts:38-52`, `applyRemoteChange`; `lib/sync/pb-pull.ts:166-188`, `reconcileDeletedTasks`

**Issue:** Exhausted queue items are marked `failed` and excluded from future pushes, but deletion reconciliation treats every queue row as pending regardless of status. A failed item therefore cannot succeed and simultaneously protects its local task from remote deletion indefinitely.

**Impact:** When another device deletes the task, the affected device retains a zombie record across realtime events and full pulls. Users see inconsistent data between devices with no automatic convergence.

**Evidence:** `isPending()` includes only status `pending`, and `recordAttemptFailure()` sets `status = 'failed'` after the retry budget. Realtime deletion uses `db.syncQueue.toArray()` and skips on any matching task at `pb-sync-engine.ts:45-48`; pull reconciliation similarly maps every queue item into `pendingTaskIds` at `pb-pull.ts:177-181`. Neither path checks status.

**Reproduction:** Insert a local task and a matching sync-queue row with `status: 'failed'`. Apply a realtime delete or run a pull whose remote index omits the task. The local task remains even though the row is not returned by `getPending()` and will not be retried.

**Recommended fix:** Share the queue's pending-status predicate with both deletion guards, and define an explicit product policy for failed rows (manual retry, discard-local, or conflict resolution). Add tests for remote deletion with both pending and failed queue states.

**Confidence:** Confirmed

#### Accessibility — Settings renders nested main landmarks and duplicate page headings

**Severity:** Medium

**Category:** Accessibility

**Location:** `components/matrix-simplified/app-shell.tsx:82-117`, outer landmark; `components/settings-page/settings-body.tsx:126-194`, nested landmark; `components/matrix-simplified/topbar.tsx:37-43`; `components/settings-page/index.tsx:22-33`

**Issue:** `AppShell` already wraps page content in `<main>`, but `SettingsBody` adds a second `<main>` inside it. The shell topbar and settings content also both render `Settings` as `<h1>`.

**Impact:** Screen-reader landmark and heading shortcuts announce an invalid nested main structure and two competing page-level headings, making page orientation and section navigation unnecessarily ambiguous.

**Evidence:** The outer main is at `app-shell.tsx:114`; `settings-body.tsx:127` renders another main before closing it at line 194. `topbar.tsx:38-43` renders the shell title as `h1`, and `settings-page/index.tsx:27-29` renders the same title again as `h1`. The runtime accessibility snapshot confirmed two main landmarks and two `Settings` level-one headings.

**Reproduction:** Open `/settings/` and inspect the accessibility tree or run `document.querySelectorAll('main, h1')`; two nested main elements and two `h1` elements are returned.

**Recommended fix:** Change `SettingsBody` to a `section` or `div` with an appropriate accessible name, and establish exactly one page-level `h1`. Add semantic-structure assertions to the page test.

**Confidence:** Confirmed

#### Accessibility — Core mobile controls miss the 44-pixel touch-target floor

**Severity:** Medium

**Category:** Accessibility

**Location:** `components/matrix-simplified/icon-rail.tsx:105-165`, mobile navigation; `components/matrix-simplified/capture-bar.tsx:149-158,208-221`; `components/task-card/task-card-header.tsx:62-71`; `app/globals.css:964-972`

**Issue:** Mobile navigation buttons are fixed at 40 by 40 pixels, the primary Add button is 36 pixels high, the capture input's interactive height is about 21 pixels, and drag handles are about 28 by 28 pixels. The coarse-pointer 44-pixel rule only applies to `.touch-target`, which these controls do not use.

**Impact:** Touch users are more likely to miss core navigation, capture, and reorder actions, especially with reduced dexterity or while using a phone one-handed. The issue affects high-frequency product controls rather than secondary decoration.

**Evidence:** The mobile rail uses `h-10 w-10` at line 156; Add uses `h-9` at `capture-bar.tsx:213`; the input has no minimum block size; and the drag button combines a 16-pixel icon with 6-pixel padding at `task-card-header.tsx:62-71`. At a 375 by 812 viewport, runtime measurements were 40 by 40 for bottom navigation, 81 by 36 for Add, approximately 144 by 21 for the input, and 28 by 28 for drag handles.

**Reproduction:** Load `/` at a 375 by 812 viewport with sample tasks and inspect `getBoundingClientRect()` for the bottom navigation buttons, capture input, Add button, and drag handle.

**Recommended fix:** Apply at least `min-h-11 min-w-11` or the shared `.touch-target` contract to mobile controls, enlarge the capture field's interactive box, and preserve spacing so expanded hit areas do not overlap. Add coarse-pointer size checks to responsive UI tests.

**Confidence:** Confirmed

#### Visual Consistency — Tablet breakpoint combines the full rail with the compact two-column matrix

**Severity:** Medium

**Category:** Visual Consistency

**Location:** `components/matrix-simplified/icon-rail.tsx:58-62`; `components/matrix-simplified/matrix-grid.tsx:50-52`; `components/matrix-simplified/app-shell.tsx:114`; `components/matrix-simplified/quadrant-pane.tsx:107-136`

**Issue:** At exactly the `md` breakpoint, the UI simultaneously reveals the 180-pixel expanded rail, switches the task matrix to two columns, and retains substantial horizontal page padding. Quadrant headings, explanatory copy, and task titles are squeezed into narrow columns even though there is no horizontal overflow.

**Impact:** Common portrait-tablet widths present visibly fragmented headers and prematurely truncated task content, reducing scanability in the application's primary decision surface. The same components look materially more coherent just below and above this width.

**Evidence:** The rail becomes visible at `md` and defaults to `md:w-[180px]`; the matrix changes to `md:grid-cols-2` at the same breakpoint; the shell adds `sm:px-9`. A 768 by 1024 browser capture showed quadrant labels/hints wrapping across several short lines and task titles truncating while the expanded rail consumed almost one quarter of the viewport.

**Reproduction:** Load a populated matrix at 768 by 1024 with the rail in its default expanded state. Compare header and task-title wrapping with 767 pixels or with the rail manually collapsed.

**Recommended fix:** Auto-collapse the rail until a larger breakpoint, delay the two-column matrix to `lg`, or use a container query that switches based on the matrix's actual remaining width. Compact or hide secondary quadrant hints when the pane width is below its readable minimum.

**Confidence:** Confirmed

## Remediation plan

### Quick wins

1. Give `OnboardingGate.close()` a real observable state transition and add a first-run dismissal regression test.
2. Replace the nested Settings main landmark and remove the duplicate `h1`.
3. Filter deletion guards through the existing pending-status predicate so failed sync rows no longer masquerade as pending.
4. Redact `pocketBaseUrl` from MCP output and reject credentials/query secrets in configured URLs.
5. Enlarge the identified mobile hit areas and align them with the existing `.touch-target` contract.
6. Update the affected development/transitive packages and verify that `bun audit` is clean or that any remaining advisories have documented, tested non-reachability.

### Deeper work

1. Standardize all modal behavior on one accessible dialog primitive with focus containment, inert background, Escape behavior, and invoker restoration.
2. Introduce a transaction boundary for task/dependency changes plus sync-queue persistence, with failure-injection coverage for rollback.
3. Make recurring completion idempotent across rapid clicks and concurrent callers using a transactional state check and deterministic recurrence identity.
4. Define a user-visible failed-sync resolution state instead of allowing permanent queue records to influence reconciliation implicitly.
5. Rework the matrix breakpoint around available container width, not only viewport width, and add screenshot coverage at portrait-tablet widths.
