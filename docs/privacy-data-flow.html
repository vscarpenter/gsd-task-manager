03Privacy claims check

Each stated guarantee, marked against what the code actually does.

Local-first Confirmed

IndexedDB is the system of record. The entire create → validate → persist → read loop is local Dexie calls with zero fetch in the path. On a fresh install with no sign-in, task content never leaves the browser.

  • Persistence is db.tasks.add/put only — crud/create.ts:41
  • Reads via local live query — use-tasks.ts:18
  • Sync is off unless explicitly enabled — config/get-set.ts:77

No server-side storage of task content Confirmed with one caveat

The shipped app has zero Next.js server touchpoints — no Server Actions, no app/api route handlers, no middleware. output: 'export' makes a server runtime structurally impossible, and the build emits no server/edge functions. The app cannot store your content because it has no server to store it on.

  • rg "use server" → 0 real hits; app/ is 10 page/layout/error files, no route.ts, no middleware.ts
  • Static export confirmed — next.config.ts:8
Caveat — the claim is scope-sensitive. When a user opts into cloud sync, full task content (title, notes, tags, subtasks) is stored as plaintext on the PocketBase backend at api.vinny.io (task-mapper.ts:54), and the MCP server reads/writes it there. That is server-side storage — on a self-hosted server, opt-in, but real. Recommend the public claim read “no server storage unless you enable sync.”

No third-party trackers Confirmed

No Google Analytics/GTM, Plausible, Fathom, Vercel Analytics, Segment, PostHog, Mixpanel, Hotjar, or Pixel in source or dependencies. Fonts are self-hosted (no runtime Google Fonts). The CSP connect-src enforces this at the policy layer.

  • Egress allowlist is 4 origins only — app/layout.tsx:43
  • Note: Sentry is present as error telemetry (not behavioral tracking) and is content-allowlist-gated; Google/GitHub are contacted only for OAuth. None profile usage.

Stated in-app: “End-to-end encrypted · zero-knowledge” Needs attention

This is the one place the UI contradicts the code. There is no application-level or E2E encryption anywhere — content is plaintext in IndexedDB and plaintext-over-TLS to the server. The marketing copy overstates the guarantee.

  • ⚠ “End-to-end encrypted sync… the server never sees your data” — false — features-section.tsx:84
  • “encrypted in transit” is TLS-true but framed to imply more — privacy-section.tsx:30
  • Stale ENCRYPTION_PASSPHRASE in MCP docs (no code reads it) — mcp-section.tsx:16
  • Secondary: route error-logger.ts:59 through the Sentry allowlist so content-exclusion is structural, not by convention.

How this was traced

  1. Six parallel read-only passes over the working tree at commit d004f1f: the write path (UI→state→IndexedDB), encryption, PocketBase sync, server touchpoints, analytics/telemetry/logging, and export/import + MCP.
  2. Load-bearing claims were spot-checked directly: rg "use server", find app -name route.* and middleware globs (server touchpoints); crypto-term grep + package.json (no encryption); and verbatim reads of the stale about-page copy.
  3. Every file : line link is pinned to d004f1f, so anchors stay accurate even as main moves on.

Runtime caveats. Zones are static-analysis truths; at runtime the red zone is entered only by user action (enabling sync, configuring MCP). “Encrypted at rest” here means application-controlled encryption — browser/OS disk encryption and TLS still apply but are outside the app’s control. The PocketBase backend is self-hosted; “server” means a server you (or the operator) run, not a third party.
