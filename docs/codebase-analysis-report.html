1. Executive Summary

6
out of 10

Overall health is solid but not green. The codebase demonstrates excellent unit-test coverage, strict TypeScript discipline, and thoughtful modular architecture. However, the E2E suite is unstable on Firefox, dependency audits expose five high-severity CVEs, and several high-impact user workflows lack end-to-end coverage. The build also carries two ESLint errors that block a clean lint run.

The score follows the project's rubric: coverage exceeds 80%, but there are multiple high-severity dependency vulnerabilities and several high-severity quality gaps. No critical data-loss or security exploit was identified in the application code itself.

Top 5 Risks

  1. 1Dependency CVEs (5 high) — undici, vite, and hono carry known high-severity vulnerabilities affecting the test runner, build tooling, and MCP server package.
  2. 2Firefox E2E instability — 10 of 12 failing E2E tests are page.reload()/page.goto() timeouts on Firefox, masking real regression signal.
  3. 3Incomplete E2E Page Object Model — only one POM class exists for 16 spec files; text-based selectors dominate.
  4. 4Security-header deployment drift — CSP/HSTS are authored in CloudFront JSON and Caddyfile, but there is no automated validation that the deployed policy matches the source.
  5. 5Missing critical-path E2E coverage — bulk operations, dashboard metrics, offline PWA behaviour, and real sync flows have no end-to-end coverage.

Top 3 Strengths

  1. 1Strong unit-test discipline — 89.5% statement coverage, 90.6% line coverage, 2,181 passing tests, and Zod-validated schemas on every data boundary.
  2. 2Modular, layer-respecting architecture — clean separation between lib/tasks/, lib/sync/, lib/analytics/, and lib/notifications/; no circular dependencies detected.
  3. 3Security-conscious code — input validation via Zod, secret masking in logger and Sentry, CSP meta tag in layout, no eval() or dangerouslySetInnerHTML, and admin API routes blocked.

Top 3 Priorities for the Next Sprint

  1. 1Stabilize Firefox E2E — replace hash-reload patterns, add missing page objects, and reduce hardcoded waitForTimeout usage.
  2. 2Patch or mitigate dependency CVEs — update undici, vite, and hono to non-vulnerable versions; verify MCP server package separately.
  3. 3Close critical-path E2E gaps — add specs for bulk operations, dashboard analytics, and offline PWA behaviour.

3.1 Standards Compliance

Lint errors block clean verification Medium High confidence
File: scripts/build-openwiki-site.cjs:23-24 · Effort: Small

A .cjs build script uses require() for gray-matter and marked, triggering @typescript-eslint/no-require-imports errors. The ESLint config does not exclude build scripts from this rule, so bun lint fails in CI.

Recommendation

Add the script to the ESLint ignore list or add a targeted override for scripts/*.cjs that allows CommonJS require because the file is intentionally CJS.

React ref warnings suppressed with doctor comments Low High confidence
File: components/task-card/task-card-header.tsx:42-49 · Effort: Small

The component reads ref.current during render to detect a just-completed transition and applies an animation class. ESLint flags it as a rule-of-hooks violation. The inline disable comment explains the intent, but the pattern is fragile under React Compiler or concurrent rendering.

Recommendation

Move the "just completed" detection into an effect or derive it from a transition timestamp; encapsulate the animation trigger in a custom hook so the justification is centralized.

Production files approach the 400-line threshold Low Medium confidence
Files: components/matrix-simplified/index.tsx (336), packages/mcp-server/src/write-ops/task-operations.ts (372), lib/db.ts (390) · Effort: Medium

None exceed the 400-line guideline, but three files are within 10–15% of the limit. The matrix-simplified index orchestrates many concerns (capture, edit drawer, dnd, search, smart views, command palette, hotkeys); the MCP task-operations file bundles create/update/complete/delete operations.

Recommendation

Extract the overlay reducer and window-event handlers from the matrix index into separate hooks; split MCP task-operations into per-operation files under write-ops/operations/.

3.2 Test Quality & Coverage

E2E Page Object Model is incomplete High High confidence
File: tests/e2e/pages/matrix-page.ts (only POM) · Effort: Medium

Only one page object exists, yet specs navigate to settings, dashboard, archive, about, install, and sync-history. Settings and dashboard tests rely on raw page.locator() and getByRole() calls, duplicating locator logic and making UI changes expensive.

Recommendation

Add SettingsPage, DashboardPage, ArchivePage, and AboutPage POMs. Move repeated selectors (e.g., sidebar sections, chart containers) out of spec files.

Heavy reliance on text-based selectors High High confidence
Files: tests/e2e/task-crud.spec.ts, recurring-tasks.spec.ts, drag-and-drop.spec.ts, search.spec.ts · Effort: Medium

Task cards are repeatedly located with page.locator("[data-testid='task-card']").filter({ hasText: title }). This couples tests to copy, breaks on whitespace/HTML entities, and is incompatible with future i18n.

Recommendation

Seed tasks via IndexedDB and capture generated IDs in test helpers, then use per-task data-testid attributes such as data-testid="task-card-${id}". Alternatively, filter by a stable child data-testid='task-title' rather than free text.

Missing E2E coverage for critical workflows High High confidence
Files: missing tests/e2e/bulk-operations.spec.ts, tests/e2e/dashboard.spec.ts, offline PWA tests · Effort: Large

The app ships bulk operations (multi-select, complete, move, tag, delete), dashboard analytics, and offline PWA behaviour, but none are exercised end-to-end. The existing dashboard test only verifies navigation, not rendered metrics or charts.

Recommendation

Add bulk-operations.spec.ts covering selection, bulk delete, and bulk move. Add dashboard.spec.ts that seeds completed tasks and asserts chart visibility and metric values. Extend pwa-install.spec.ts with context.setOffline(true) and a cache reload.

Hardcoded waitForTimeout is pervasive Medium High confidence
Files: tests/e2e/pages/matrix-page.ts, tests/e2e/helpers/test-helpers.ts, multiple specs (~56 occurrences) · Effort: Medium

Specs and helpers wait fixed durations for rendering, animations, and post-click settling. These waits slow the suite and introduce flakiness when CI machines are slower than the local dev environment.

Recommendation

Replace fixed waits with expect(...).toBeVisible(), waitForLoadState("networkidle"), or component-specific animation-end signals.

Hardcoded IndexedDB schema version in E2E helpers Medium High confidence
Files: tests/e2e/recurring-tasks.spec.ts:29, tests/e2e/drag-and-drop.spec.ts:79, :138, :194 · Effort: Small

Tests open IndexedDB with DB_VERSION = 140, while the actual Dexie schema version in lib/db.ts is 14. The hardcoded value will cause silent breakage when the schema is bumped.

Recommendation

Export the schema version from lib/db.ts and import it in E2E helpers, or read the version from the Dexie instance directly.

Excellent unit-test coverage and structure Strength High confidence
Files: tests/data/, tests/ui/, tests/sync/ · Effort: N/A

Unit and integration tests follow AAA structure, use realistic fixtures, and cover sync engine, queue, auth, CRUD, import/export, and analytics. Coverage exceeds the 80% floor across all dimensions and reaches 100% in several analytics modules.

3.3 Security & Supply Chain

Multiple high-severity dependency CVEs High High confidence
Files: package.json, packages/mcp-server/package.json · Effort: Small–Medium

bun audit reports 16 vulnerabilities, including 5 high: undici (TLS validation bypass, DoS, SOCKS5 proxy reuse), vite (server.fs.deny bypass on Windows), and hono (CORS wildcard reflection, body-limit bypass). These affect dev tooling, test runtime (jsdom), and the MCP server package.

Recommendation

Update vite to a patched 8.x release, bump jsdom (pulls newer undici), and bump the MCP server @modelcontextprotocol/sdk dependency to a version that pulls a patched hono. Re-run bun audit before committing.

Security headers exist in code but deployment validation is manual Medium High confidence
Files: cloudfront/response-headers-policy.json, docker/Caddyfile, app/layout.tsx · Effort: Small

Production CSP/HSTS/COOP/Permissions-Policy are authored in CloudFront JSON and the Caddyfile, and a CSP meta tag is rendered in layout. However, there is no automated test or CI step that verifies the deployed headers match the source. The May 2026 security review found the Caddyfile missing CSP; that has since been fixed, but drift risk remains.

Recommendation

Add a post-deploy smoke test or CI lint step that parses cloudfront/response-headers-policy.json and asserts the required directives are present and that unsafe-eval is absent in production.

Security controls are well-implemented in code Strength High confidence
Files: lib/capture-parser.ts, lib/task-links.ts, lib/share-capture.ts, lib/logger.ts, lib/sentry.ts, docker/Caddyfile · Effort: N/A

No eval(), Function(), or dangerouslySetInnerHTML exists in project code. URLs are validated with allow-lists and reject userinfo/password. Logger and Sentry sanitize tokens, passwords, and auth headers. The Caddyfile blocks /api/collections/_superusers/* and /api/admins/*. The previously reported hard-coded api.vinny.io fallback has been removed: lib/env-config.ts now falls back to window.location.origin for unknown hostnames.

3.12 E2E Test Health

Firefox reload/goto timeouts are the dominant failure mode High High confidence
Files: tests/e2e/recurring-tasks.spec.ts, drag-and-drop.spec.ts, settings-sections.spec.ts, sync-states.spec.ts · Effort: Small

All 10 Firefox failures are Test timeout of 30000ms exceeded on page.reload() or page.goto(). The console shows ChunkLoadError warnings during the run, suggesting the dev server or service worker is serving stale chunks. This is likely an environment/test-setup issue rather than application bugs, but it makes the Firefox project unusable as a regression gate.

Recommendation

Clear service-worker caches before each test, avoid page.reload() on Firefox (use page.goto() with the cleaned URL), and consider a dedicated test build (bun run build + static server) instead of the dev server for E2E to eliminate chunk-load races.

WebKit dependency chip timeout Medium High confidence
File: tests/e2e/task-dependencies.spec.ts:17 · Effort: Small

The single WebKit failure waits for a dependency chip to become visible after selecting a suggestion. This may be a timing/animation issue specific to WebKit or a selector that is too strict.

Recommendation

Add an explicit wait for the suggestion list to close before asserting the chip, and use expect(...).toBeVisible() with a longer timeout rather than a raw waitFor.

5. Methodology & Limitations

What was inspected

  • Static analysis: source files in app/, components/, lib/, tests/, packages/mcp-server/, scripts/, docker/, and configuration files.
  • Verification: bun typecheck, bun lint, bun run test -- --coverage, bun audit, and a full Playwright E2E run against Chromium, Firefox, and WebKit.
  • Design system: Inkwell 3.0.1 instructions were fetched from raw.githubusercontent.com/vscarpenter/inkwell/main/agent-instructions.md and applied to the report stylesheet (system fonts, 1.5px borders, single indigo accent, ivory background).
  • Excluded: node_modules, lockfiles, build artifacts (.next, out), generated coverage/test-results, and editor/agent configuration directories.

Limitations

  • Single E2E data point: pass/fail counts reflect one local run. No historical CI trend was available, so flakiness is inferred from failure patterns rather than measured.
  • Runtime profiling was not performed: performance findings are based on code inspection and test coverage, not on bundle-size budgets or production traces.
  • Cloud-hosted infrastructure was not tested: security headers were inspected in source files but not validated against a live CloudFront distribution or deployed Caddy instance.
  • Dependency exploitability: CVEs are reported from bun audit; whether each vulnerability is reachable in the app's actual runtime paths was not verified by hand.
  • Excluded generated files: the report excludes coverage JSON and Playwright JSON output files because they are gitignored and therefore not directly readable by the analysis tooling.
