Detailed Findings

18 findings

1. Standards Compliance

▶ Compliant File size, naming, and structure conventions Confidence: High

All 188 source files adhere to the 350-line maximum. PascalCase for components/types, camelCase for functions, kebab-case for filenames. @/ alias used consistently for imports. Tailwind classes grouped by layout → spacing → color.

▶ Medium 3 functions exceed 40-line limit Effort: Small

Files: components/matrix-simplified/edit-drawer.tsx (~50 lines), components/settings-page/index.tsx (~55 lines), lib/db.ts (migration chains).

Recommendation: These are UI orchestration and schema definition contexts — acceptable exceptions per ADR 0011. Consider extracting form field groups in edit-drawer to reduce component body length.

2. Test Quality & Coverage

▶ Strength Unit test coverage exceeds all targets Confidence: High

Statements: 84.81% (target: 80%). Lines: 85.70% (target: 80%). Functions: 81.99% (target: 80%). Branches: 78.62% (target: 75%). 1,857 tests pass with 0 failures across 114 test files.

Patterns: Proper Arrange-Act-Assert structure. Mock IndexedDB with fake-indexeddb. Behavior-based test names (should_return_...). Independent tests with no shared mutable state.

▶ High Zero coverage for lib/tasks.ts and lib/confetti.ts Effort: Medium

lib/tasks.ts shows 0% coverage across all metrics. This appears to be a barrel re-export superseded by lib/tasks/ modules. lib/confetti.ts (canvas-confetti wrapper) has 0% coverage.

Recommendation: Verify lib/tasks.ts is not dead code; if it is, remove it. For confetti, add a simple mock-based test verifying the function is callable.

▶ Medium Vitest config picks up Playwright spec files Effort: Small

Running bun run test reports 5 “failed” files — all are Playwright .spec.ts files incorrectly matched by Vitest. This inflates the failure count in CI.

Recommendation: Add exclude: ['tests/e2e/**'] to vitest.config.ts:

// vitest.config.ts
exclude: [...configDefaults.exclude, 'tests/e2e/**']

3. Security & Supply Chain

▶ Strength Zero audit vulnerabilities, zero XSS sinks Confidence: High

bun audit reports zero vulnerabilities across all 51 dependencies. No dangerouslySetInnerHTML. No hardcoded secrets. CSP headers configured in Docker Caddyfile and layout.tsx. PocketBase auth uses SDK-managed JWT with auto-refresh.

▶ Medium .parse() on user-input paths should be .safeParse() Effort: Small

Affected files:

  • lib/tasks/crud/update.ts:28 — taskDraftSchema.parse(nextDraft)
  • lib/notifications/settings.ts:33 — notificationSettingsSchema.parse(settings)
  • lib/notifications/settings.ts:53 — notificationSettingsSchema.parse(updated)

.parse() throws on invalid input, aborting the operation. The project standard requires .safeParse() with explicit error handling on all user-input paths.

▶ Medium Hard-coded PocketBase URL fallback Effort: Small

File: lib/env-config.ts

The env config falls back to https://api.vinny.io when NEXT_PUBLIC_POCKETBASE_URL is not set. Self-hosted deployments may silently connect to the wrong instance.

Recommendation: Remove the hardcoded fallback. Require the env var explicitly and throw during initialization if missing.

4. Architecture

▶ Strength Clean dependency graph, no circular imports Confidence: High

Unidirectional dependency DAG throughout. lib/sync/ (20+ modules) properly layered. lib/tasks/ splits CRUD into 10 focused files. lib/analytics/ maintains 5 side-effect-free modules. All 56 database accesses funneled through getDb().

5. Maintainability

▶ Compliant Excellent module boundaries and file sizes Confidence: High

Zero files exceed 350 lines. Zero commented-out code blocks. Zero TODO/FIXME in production source (tracked in tasks/todo.md). All 3 any type usages include justification comments.

6. Dependencies

▶ Healthy Well-managed with minimal risk Confidence: High

27 production dependencies, all current versions. bun audit: 0 vulnerabilities. Each dependency serves a clear purpose with no duplicates.

Note: @playwright/test uses ^1.60.0 (floating range) in devDependencies — acceptable for dev-only tooling but worth pinning in CI for reproducibility.

7. Performance

▶ Optimized No N+1 queries, proper memoization, virtual scrolling Confidence: High

Batch operations (db.tasks.toArray()) used throughout. React Compiler enabled for automatic memoization. 8 memoized metrics in dashboard. @tanstack/react-virtual for large lists. No synchronous I/O on hot paths.

8. Error Handling & Observability

▶ High No global unhandledrejection listener Effort: Small

No window.addEventListener('unhandledrejection', ...) handler. The realtime event handler in lib/sync/pb-realtime.ts:83 has no try/catch — errors bubble as unhandled rejections that fail silently.

Recommendation: Add a global listener at the app root that logs via createLogger('ERROR_BOUNDARY') and shows a toast.error().

▶ Medium No server-side error tracking service Effort: Medium

The structured logger (24 context types, secret sanitization, correlation IDs) outputs only to console.*. Production errors are only visible if a user opens DevTools.

Recommendation: Integrate Sentry, LogRocket, or a self-hosted alternative. The logger architecture already supports this via the placeholder at lib/error-logger.ts:37.

▶ Strength Comprehensive structured logging with 24 contexts Confidence: High

43 files use createLogger(). Only lib/logger.ts itself makes raw console.* calls. Secret sanitization masks tokens, API keys, passwords, JWTs. Error boundary wraps the entire app in layout.tsx.

9. Documentation & Onboarding

▶ Excellent Comprehensive documentation ecosystem Confidence: High

11 ADRs. CLAUDE.md for AI-agent guidance. coding-standards.md v14. DATABASE_ARCHITECTURE.md for ERD. .github/copilot-instructions.md for VS Code integration. Local dev: bun install && bun dev.

10. Tech Debt

▶ Low Dead code detection not automated Effort: Small

Adoption of knip or ts-prune tracked in tasks/todo.md but not implemented. Manual audits removed 6 dead files in v9, but ongoing detection relies on human review.

Recommendation: Add knip to CI for automated dead code and unused dependency detection.

▶ Low Command palette not wired into v9 shell Effort: Medium

The command palette (components/command-palette/) has 17 passing unit tests but is not rendered in the v9 app shell. Code is maintained but not exercised in production.

11. Configuration & Type Safety

▶ Compliant TypeScript strict mode, typed routes, Zod validation Confidence: High

TypeScript strict mode enabled. Next.js typed routes configured. All data types have Zod schemas. Constants centralized in lib/constants/. No magic numbers.
