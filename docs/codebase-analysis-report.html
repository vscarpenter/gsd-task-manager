Top 5 Risks

The prior #1 risk — coverage-padding (F2.1) — has been resolved this cycle and dropped off the list.

  1. Core flows have zero E2E coverage. Recurring tasks, task dependencies, and drag-and-drop — all central features with unit tests only — are never exercised end-to-end across browsers. High
  2. Docs advertise deleted features and are two majors stale. README.md markets Command Palette, Smart-View pinning, and Batch Operations (all removed in v9 per ADR 0011) with ✅ checkmarks, and declares "Current Version: 7.8.1" against a package.json at 9.3.6. High

Only two moderate transitive CVEs remain, neither exploitable in the browser bundle.

MediumFlaky matrix-navigation tests — confirmed across two runsF12.4

Confidence: High · Location: tests/e2e/matrix-navigation.spec.ts (settings + dashboard nav) · Effort: Small

Now confirmed flaky. The first run failed 1 test on WebKit (a Radix data-state="closed" vs {null} hydration diff on the theme toggle); this re-run failed 2 different nav tests on Chromium, with WebKit green. A failure that moves browsers and count between runs is a timing flake, not an engine-specific bug — and the suite's retries: 0 means each flake surfaces as a hard failure. The root cause is the waitForTimeout reliance flagged in F12.2.

Recommendation: Replace the matrix-navigation sleeps with await expect(locator).toBeVisible()-style assertions (per F12.2); do not paper over it with retries. The fix removes both the flake and the ambiguity.

Methodology & Limitations

What was inspected

  • This is a re-run on branch test/decouple-smart-views-coverage-padding after the F2.1 remediation. Because that work touched only test files (no lib//components//app/ production code), the empirical data (coverage, unit + E2E runs) and the test-quality dimension were freshly measured; the security, architecture, dependency, performance, observability, config, and documentation findings were carried forward from the prior run after verifying their underlying code is byte-identical.
  • Production source: lib/, components/, app/, scripts/, packages/mcp-server/src/ (248 source files).
  • Tests: Vitest unit/integration suites and Playwright E2E specs.
  • Config & CI/CD: vitest.config.ts, playwright.config.ts, package.json, deploy workflows, and the CloudFront response-headers policy. Note: this is a static-export PWA — "infrastructure as code" is limited to scripts/deploy-*.sh and the CloudFront JSON; there is no Terraform/CDK.
  • Live evidence (this run): bun run test -- --coverage (1,849 passing tests; coverage-summary.json), Playwright across Chromium/Firefox/WebKit, bun audit, bun outdated, madge, and targeted greps. Prior-run findings were gathered by four parallel read-only analysis agents; every High-severity claim was independently re-verified against source.

What was NOT inspected / caveats

  • next build was not run → bundle-size and code-splitting findings (F7.2) are inferred, not measured; flagged Low confidence.
  • Performance was assessed statically, not by runtime profiling; F7.x are scaling notes, not observed regressions.
  • Cyclomatic/cognitive complexity averages were not computed (no complexity tool was run). File and function size (Dimensions 1 & 5) are used as a structural proxy; the metrics dashboard reports size counts in place of complexity averages.
  • E2E was run twice across this cycle (prior run + this re-run), each with retries: 0 locally. The two runs failed different matrix-navigation tests on different browsers, which is what upgraded F12.4 from "undeterminable" to "confirmed flaky". Still local-only — no CI history for duration trends.
  • No duration "trend" is available — a single run is one data point and there is no historical/CI baseline; absolute durations are reported instead (per the corrected prompt).
  • Severity was normalized across four independently-calibrated agents to one rubric. Nothing is graded Critical because the build and test suites pass, coverage clears all thresholds, and no data-loss, secret, or XSS was found. Two agent-assigned "Critical" labels (the duplicate test directory and coverage-padding files) were normalized to Medium and High respectively.
  • Authenticated sync & live PocketBase behavior were reviewed by code reading only — no credentialed live-server test was performed.
  • MCP server tests are excluded from the main Vitest run (separate workspace) and were not executed for this report.
  • Design system: the Inkwell spec was fetched to confirm tokens, but styling is applied via the repo-local docs/css/inkwell.css (already present and conformant) rather than re-downloading the stylesheet bundle.
