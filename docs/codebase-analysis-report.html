▶ Security & supply chain 1 High2 Med2 Lowclean core
HIGHH1Unbounded client_updated_at enables last-write-wins poisoning
Confidence: High · Effort: Medium · Location: lib/sync/task-mapper.ts:102, lib/sync/pb-pull.ts:90, lib/sync/pb-sync-engine.ts:74, lib/sync/pb-push.ts:171

client_updated_at — the LWW authority — is validated only as z.string() (no range) and originates from the writing device's own new Date().toISOString(). All three comparison sites are unclamped (isRemoteNewer only guards NaN). A device with a skewed clock (e.g. year 2099) — or a user POSTing a forged timestamp to their own record (the updateRule pins owner, not field values) — permanently wins every future comparison, making that task un-editable from any other device.

The existing "year-3000 clamp" test (tests/data/sync/pb-pull.test.ts:80) only covers the cursor (server-stamped updated field), never the LWW client_updated_at path — so a regression here is untested.

Fix: clamp client_updated_at to min(value, now+5min) at comparison or at pull-validation in task-mapper.ts, mirroring the cursor convention.
Falsifies this: PocketBase enforcing server-side date-range validation on client_updated_at writes — not present in the setup script, which defines only text-length options for that field.
HIGHH10Raw error message may reach Sentry via default console breadcrumbs, bypassing the allowlist
Confidence: Medium ASSUMPTION · Effort: Medium · Location: lib/logger.ts:187, lib/sentry.ts:15, lib/sync/retry-manager.ts:36

Logger.error() writes the raw error.message to console.error (sanitization masks by key name, so the errorMessage value passes through). Documented in-code: PocketBase 4xx bodies echo submitted task titles. initSentry sets no beforeBreadcrumb / defaultIntegrations:false, so the SDK's default console-breadcrumb integration may attach that unmasked content to the next captured event — defeating the otherwise-correct 19-key allowlist.

Fix: mask errorMessage/stack in the console path (as maskError() already does for the Sentry path), and set beforeBreadcrumb to drop/truncate console breadcrumbs.
Falsifies this: pulling a real Sentry event for a sync error and finding no console breadcrumb carrying a task title — not verifiable without dashboard access this session.
MEDM9Failed update logs the raw task content to the console
Confidence: High · Effort: Small · Location: lib/tasks/crud/update.ts:37

On a failed update, logger.error(..., { taskId: id, updates }) prints the raw updates object (title/description/tags) to the browser console — directly contradicting the sibling policy comment in create.ts:30 ("Never log the raw input — it holds the task title/description"). Local-console-only; the allowlist keeps it off Sentry.

Fix: drop updates from the log context, matching create.ts/delete.ts.
LOWMCP queries rely solely on server-side owner rules; long-lived token has no rotation
Confidence: Medium ASSUMPTION · Effort: Small · Location: packages/mcp-server/src/write-ops/helpers.ts, tools/list-tasks.ts:56

MCP reads/writes filter by task_id only, depending entirely on PocketBase's owner = @request.auth.id rules with no client-side belt-and-braces owner filter — correct-by-construction today but a single point of failure if server rules drift. The MCP auth token is a long-lived JWT with no rotation beyond manual re-auth.

Fix: add a defensive client-side owner filter to MCP queries; document token-rotation guidance.
LOWMCP-vs-webapp field-mapper default inconsistency & unallowlisted MCP Sentry context
Confidence: High · Effort: Small · Location: packages/mcp-server/src/types.ts:171 vs lib/sync/task-mapper.ts:72; packages/mcp-server/src/utils/sentry.ts:52

MCP maps missing notifyBefore/estimatedMinutes to 0 while the webapp maps them to null — a "no estimate" vs "0-minute estimate" fidelity gap across two mapper implementations. Separately, the MCP Sentry helper forwards context verbatim with no allowlist (no risky call sites today, but a foot-gun).

Fix: align the two mappers on null; apply the lib/logger.ts allowlist pattern to the MCP Sentry helper.

Process

Direct metric collection was reconciled against five parallel read-only subagents (architecture/performance, security/data-integrity, test-quality, error-handling/observability, docs-drift/counter-view), each required to open and cite file:line before asserting. Every High finding was independently re-verified against the cited lines by the lead before inclusion, and two agent-proposed "Critical" findings were downgraded to High after that check (H2, H5). Cross-dimension findings were deduplicated to one primary dimension with cross-references.

Confidence-calibration rule

Confidence is High only for claims backed by observed code at a cited line. It is lowered to Medium and tagged ASSUMPTION whenever a claim rests on something not observable from this repo — runtime behavior, live server state, or SDK-default behavior. Severity is never inflated to force attention, and no metric is presented as measured when it was inferred (the function-length heuristic is explicitly labeled approximate).
