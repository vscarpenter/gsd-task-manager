The idea

Everything between the gates is automated. Everything at the gates is judgment.

A change starts as a GitHub issue written like a spec. A builder agent claims it, drafts a plan sized to its risk, and stops for approval. On your nod it builds to versioned coding standards, opens a pull request, and a reviewer agent checks it alongside CI. When it's clean you approve the release after testing the running software. A third agent works nights, repairing the mechanical failures that pile up on open PRs. The two approvals are the only human footprint — and they sit exactly where a caught mistake is cheapest.

  • Risk tiers scale the work. A docs or chore change is auto-approved past Gate 1 and built in one pass; a feature or risky change gets the full plan and a mandatory human stop.
  • Issues route back to the builder. If the reviewer or CI finds a problem, unresolved threads block release-ready and the merge until they're addressed.
  • The builder never merges. It opens a PR and stops; the pipeline lands it only after Gate 2.
Builder state machine

Labels are the durable memory between non-blocking runs

needs-triage → ready-for-agent → plan:pending → plan:approved → agent:building → PR opened
plan:revise you request changes → re-plan → back to plan:pending
ready-for-human ambiguous / needs judgment → escalate with a written reason

Approve or request changes are both label swaps — two taps on the GitHub mobile app. The runner is stateless; the laptop can sleep between runs without losing a thing.

  • Verify before submit. A fix is only offered if it makes the failing check actually pass locally — a plausible-but-wrong fix never leaves the machine.
  • Fixes re-enter the same gate. Every repair is a fresh PR that faces CI and review; the night shift never merges and never touches a branch it didn't create.
  • It files its own incident report to a control issue each run — what it fixed, escalated, or skipped, plus any anomaly it noticed about itself.

No single actor can move a change from idea to production alone. The builder that writes the code can never approve it; the reviewer can't deploy; the night shift can do neither. That control is borrowed from decades of enterprise thinking — for a solo developer it can't mean separate people, so it means separate powers.

Why the gates sit where they do

Placed on the cost curve of catching a mistake

◆ Gate 1 · Plan

A bad plan caught here costs minutes. The same plan caught after implementation costs hours of agent work, a review cycle, and tests that faithfully verify the wrong behavior. Plan review is the cheapest quality control in the pipeline — so meaningful changes get a mandatory stop, and risk tiers keep it from drowning in dependency bumps.

◆ Gate 2 · Release

By now the plan was approved, the standards held, a reviewer signed off, and CI passed. The final check is upgraded from trust to evidence — you approve software you watched run, not a diff you skimmed. The rule is absolute: no rollback path, no approval. The deploy is then proven with smoke tests, not assumed.

The learning loop

Telemetry keeps it honest; the standards are the interface

When an agent makes a category of mistake, the fix goes into the versioned coding standards, not just the code — so every future build inherits the correction and each failure is paid down once. A weekly job measures where the pipeline drags, and audit findings flow back into the standards the same way.

Safety

Every autonomous loop has a one-tap kill switch

builder:paused triage:paused Add the label to the relevant control issue and the next run posts one line and exits — no code change, no SSH. A standing audit spot-checks roughly one in ten merged PRs by hand; each agent keeps its autonomy only while those stay clean.
How it was built

Five cycles, each spec → plan → tested build → PR

  1. A
    The Contract
    An enforced GitHub issue form + risk:* labels, with an auto-labeler. Raises the floor before any agent sees the work.
  2. B
    Builder + Gate 1
    A local scheduled routine that claims a contract, writes a risk-scaled plan, stops for a label-swap approval, then builds and opens a PR.
  3. C
    Review + Gate 2
    A deterministic release-ready signal and an evidence job that hands the approver a preview link and a real rollback command. CI was also made a required check.
  4. D
    The Night Shift
    A nightly routine that triages failing checks on the fleet's own PRs — fixing the mechanical, escalating the rest, never merging.
  5. E
    Telemetry + standards loop
    A weekly collector for the four metrics, a living dashboard issue, and the bridge that turns audit findings into standards rules.
Proven live

The pipeline shipped its own first change, unattended.

A risk:chore contract was filed and auto-labeled. With no further input, the builder claimed it, auto-approved the plan, built the file to spec on a claude/* branch, and opened a pull request — which then flowed through CI, the reviewer, and release-ready before it could land. The first live run also surfaced and fixed a real deployment bug (a stripped launchd PATH), caught in seconds because the wrappers log their errors instead of swallowing them.
