
No `border-l-4`, `border-l-[3px]`, or `overdue-task` appears in any dev-compiled chunk.

**Service Worker stale-cache risk:**

The SW is registered even in dev mode (`components/pwa-register.tsx`). The SW uses cache-first for `/_next/static/*` assets (the `gsd-immutable-v1` cache). This means a browser that loaded the old chunks before the change *could* serve stale CSS from the immutable cache.

However:
- The `CACHE_VERSION` in the committed `public/sw.js` is `9.4.1`. The working-tree version is `9.4.8` (an unstaged bump). A browser visiting the dev server would get `sw.js` from the live filesystem (served network-first for HTML, and sw.js is not in the immutable cache), so any visiting browser would see the current SW version.
- Turbopack dev-mode assigns **new content-addressed chunk filenames** on every compilation cycle. The dev chunk is `Projects_gsd-taskmanager_011mdy8._.js` — a different filename from anything a prior SW would have cached. The immutable cache entries for old filenames would never be served for the new chunk names.
- **Conclusion:** The stale-chunk trap the user described is not currently active. The Turbopack dev server's content-hashed chunk naming means a new compilation cycle generates a new URL, bypassing any cached old chunk automatically.
